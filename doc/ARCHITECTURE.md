# CADette Architecture

This document is for **contributors** — it maps the codebase, explains how the
major pieces fit together, and gives extension guides for the most common kinds
of change. For *using* the command language, see
[SCRIPTING.md](SCRIPTING.md); for contribution workflow, see
[CONTRIBUTING.md](../CONTRIBUTING.md).

## Technology stack

- **Java 25**, built with **Maven**.
- **jMonkeyEngine 3 (jME3)** — the 3D engine and application framework. The app
  *is* a jME3 `SimpleApplication`.
- **Lemur** — an in-engine UI toolkit. Lemur draws its widgets inside the jME3
  scene and routes input through jME3's own `InputManager`, which is why the UI
  is consistent across platforms (a Swing-based UI was abandoned earlier for
  exactly this reason).
- **ANTLR 4** — generates the lexer and parser for the command language.
- **JTS (Java Topology Suite)** — 2D boolean geometry, used to apply cutouts
  before meshing.
- **Apache PDFBox** — cut-sheet PDF export.
- **Lombok** — boilerplate reduction.

See `pom.xml` for exact versions and the full dependency set.

## Source layout

Production code lives under `src/main/java/app/cadette/`, roughly 70 classes in
six packages:

| Package | Responsibility |
|---------|----------------|
| `app.cadette` | Application entry point, 3D scene management, orbital camera, selection, cut-sheet rendering/export, units, build info |
| `app.cadette.command` | ANTLR-driven command execution, the visitor, and every undoable `Action` |
| `app.cadette.model` | The domain model — parts, assemblies, joints, materials — plus geometry/meshing, cut-sheet packing, and templates |
| `app.cadette.lemur` | The Lemur UI shell — panels, widgets, input handling |
| `app.cadette.theme` | Theme registry and YAML-driven Lemur styling |
| `app.cadette.prefs` | Preferences persistence (`~/.cadette/preferences.yaml`) |

Other trees:

- `src/main/antlr4/app/cadette/command/` — `CadetteCommandLexer.g4` and
  `CadetteCommandParser.g4`.
- `src/main/resources/templates/standard/` — bundled `.cds` templates.
- `src/main/resources/themes/` — bundled `.cdt` theme files.
- `src/main/resources/default-preferences.yaml` — default lumber prices and
  other shipped preference defaults.
- `src/main/resources/build-info.properties` — version/commit/timestamp,
  filled in at build time.
- `src/test/java/app/cadette/` — roughly 50 headless test classes.

## Application lifecycle

`CadetteApp` is the entry point. It extends `SceneManager`, which extends jME3's
`SimpleApplication`. Launch order:

1. `CadetteApp.main()` builds `AppSettings` (1280×800, resizable, 4× MSAA,
   gamma correction **off** so theme hex colors render 1:1) and starts jME3.
2. `SceneManager.simpleInitApp()` sets up the 3D scene — disables the default
   fly-cam, attaches the orbital `CameraController`, adds the axis display and
   lighting.
3. `CadetteApp.simpleInitApp()` then wires the rest:
   - unbinds the default Escape-to-exit mapping;
   - creates the `CommandExecutor` and loads templates;
   - creates the `SelectionManager`;
   - initializes Lemur (this *must* happen after the GL context exists), loads
     themes via `ThemeRegistry`, and applies the persisted active theme;
   - attaches `LemurAppState`, which builds the panels;
   - gives `CameraController` an input blocker so camera gestures don't fire
     when the cursor is over a UI panel.

> A `~/.cadette/startup.cds` auto-run is *not* currently wired —
> `CommandExecutor.runStartupScript()` exists but nothing invokes it.

## The command pipeline

Everything the user does — typed or generated by the UI — flows through one
path, so undo/redo and scripting work uniformly.

```
command text → CommandExecutor.execute()
             → ANTLR lexer + parser  → parse tree
             → CommandVisitor        → mutates the scene, builds an Action
             → Action pushed onto the undo stack
```

`CommandExecutor` owns the pipeline. Two kinds of input are intercepted
*before* ANTLR sees them:

- **Template definitions** — once `define <name> …` is seen, the executor
  enters recording mode and buffers lines until `end define`, then parses the
  collected body as a `templateBody` and registers it.
- **Top-level control flow** — `if … then` / `for $i = …` at the REPL collect
  their statements until `end if` / `end for`, then parse as control-flow
  nodes. (Inside a template body these are already first-class grammar
  productions.)

`CommandVisitor` walks the parse tree, dispatching one method per command. It
mutates the scene and constructs an `Action`.

### Undo/redo

Every scene-changing command produces an `Action` implementing `UndoableAction`
(in `app.cadette.command` — `MoveAction`, `CutAction`, `JoinAction`,
`CreatePartAction`, and so on, including `*AssemblyAction` variants for
assembly-level operations). `CommandExecutor` keeps an undo stack and a redo
stack; a new action clears the redo stack.

A `run` of a script — and a template instantiation — buffers all of its inner
actions and wraps them into a single `ScriptRunAction`, so one `undo` reverts
the whole script atomically. A `suppressUndo` flag keeps internal work off the
stack entirely.

## Domain model (`app.cadette.model`)

**`Part`** — one piece of material. Holds its name, `Material`, cut width and
height, position, grain requirement, and its lists of cutouts and keeps. Parts
use a **corner-origin** model: the position is the part's minimum corner, and
the part extends in +X/+Y (the face) and +Z (thickness). Rotation pivots about
that corner — this matches how a woodworker thinks about orienting a board.

**`Assembly`** — a named group of part names; what a template instantiation
produces.

**`Joint`** — a **sealed interface** with one record per joint kind: `Butt`,
`Dado`, `Rabbet`, `PocketScrew`, `CountersunkScrew`, `Glue`. Each record owns
its own behavior — geometry inference (dadoes and rabbets infer a cutout in the
receiving part), validation, and BOM contribution (pocket/countersunk screws
contribute fasteners; glue contributes a glue-up line). Each joint also carries
a `SourceLocation` for error attribution. `JointType` is the parallel enum used
by the parser and for compatibility checks. `JointRegistry` stores every joint
in the scene.

**`Material`** — a catalog entry: slug, display name, `MaterialType` (plywood,
hardwood, stone, metal, …), `MaterialKind` (sheet good, solid lumber, slab,
hardware), thickness, sheet dimensions, and — for dimensional lumber — the
cross-section width and the set of standard stock lengths. `MaterialCatalog` is
the singleton holding the shipped materials.

## Geometry and meshing

A `Part` carries a list of **`Cutout`s** and a list of **keeps**. `Cutout` is a
sealed interface: `Rect`, `Circle`, `Polygon`, `Spline`, and `Curve` (cubic
Bézier). Every cutout carries an optional `depthMm` (null = through-cut,
positive = a pocket) and a `Face` (which face the pocket opens on). A `cut`
removes the shape; a `keep` retains only what is inside it.

`PartMeshBuilder` turns a part plus its cutouts into a jME3 `Mesh`:

1. Each cutout shape becomes a **JTS** polygon — rectangles directly, circles
   sampled to a polygon, splines tessellated, Bézier curves flattened via de
   Casteljau.
2. Through-cuts are subtracted from the panel rectangle with JTS boolean
   difference.
3. Pockets partition the panel into regions of constant Z; a region whose front
   and back pockets meet has no material and is dropped.
4. Each region is triangulated (via JTS) and emitted as top/bottom faces plus
   vertical walls.

## Cut sheet and packing

- **`CutListGenerator`** produces the cut list — entries grouped by material,
  each with dimensions, grain, and machining operations (sourced from joints
  where the part is the receiver, and from the part's cutouts) — plus BOM and
  fastener entries.
- **`SheetLayoutGenerator`** groups parts by material kind and routes them.
  Sheet goods go to the packer. Dimensional lumber takes a separate path that
  tries each standard stock length and picks the cheapest mix by dollar cost,
  using lumber prices from preferences. Slabs and hardware are not packed.
- **`GuillotinePacker`** does guillotine bin-packing — every cut runs
  edge-to-edge, matching what a saw actually does — honoring grain direction
  and kerf. It returns one `SheetLayout` per sheet.
- **`CutSheetRenderer`** draws a `SheetLayout` to a texture for the Cut Sheet
  panel; **`CutSheetExporter`** writes PDF/PNG/JPEG; **`CutListExporter`**
  writes CSV.

## Templates

`TemplateRegistry` holds templates keyed by case-insensitive name; names are
identifier segments joined by `/` (`standard/cabinets/base_cabinet`).
`CommandExecutor.loadTemplates()` loads the bundled templates from the
classpath (`/templates/`, walked recursively for `.cds` files) and then any
user templates under `~/.cadette/templates/`.

A `Template` keeps its parameter names, aliases, and default expressions, its
raw body lines, and the parsed `templateBody`. The `using` command scopes bare
template names against one or more namespaces.

Instantiation (`visitCreateTemplateCommand`): resolve the name, evaluate
parameter values (defaults may reference earlier parameters), create an
`Assembly` to collect the parts, push a naming-prefix context, walk the parsed
body, and record the whole thing as one `ScriptRunAction`.

Expression evaluation — arithmetic, comparisons, `min`/`max` and the math
functions, `$variable` references, and unit-suffixed literals — is performed by
the visitor as it walks expression nodes, against the executor's scope stack.

## UI layer (`app.cadette.lemur`)

`LemurAppState` is a jME3 `AppState` that builds the panel tree: **Parts** and
**Properties** on the left, the **Cut Sheet** beside the 3D viewport, and the
**Command** panel along the bottom. Panels live in resizable splitters and
draggable tab hosts; splitter ratios and active tabs persist to preferences.

`SelectionManager` is the single source of truth for what's selected. It fires
events; the 3D view and the Parts panel both listen, so selection stays
synchronized. Clicking a part selects its assembly, clicking again drills into
the part.

`CameraController` is the orbital camera: **right-drag orbits**, **left-drag
pans**, the **wheel zooms**. Its input blocker suppresses gesture *starts* when
the cursor is over a panel (an in-progress drag keeps tracking).

`ThemeRegistry` loads `.cdt` theme files (YAML) from `resources/themes/` and
`~/.cadette/themes/`, supports theme inheritance, and applies the chosen theme
into Lemur's `Styles`. `Preferences` (`app.cadette.prefs`) is a YAML-backed
singleton at `~/.cadette/preferences.yaml`, deep-merged over the bundled
`default-preferences.yaml`, with writes auto-saved.

## Build and version info

`BuildInfo` reads `build-info.properties` (version, git commit, timestamps),
which Maven fills at build time via the git-commit-id plugin and resource
filtering. The `show about` command surfaces it for bug reports.

`UnitSystem` is the unit enum. **All geometry is stored internally in
millimeters**; `UnitSystem` converts only for display and input, so a change of
units never touches stored coordinates.

## Testing

Tests run headless — `HeadlessTestBase` starts jME3 without a display — so the
full suite (~50 classes, 450+ tests) runs in CI and on any dev box with
`mvn test`. Add tests alongside any new behavior or bug fix.

## Extending CADette

### Add a material

Add an entry to the default catalog in `MaterialCatalog`. Pick the right
`MaterialType` and `MaterialKind` — `MaterialKind` decides whether the packer
treats it as a sheet good, dimensional lumber, a slab, or hardware.

### Add a joint type

Joint kinds are resolved by *name* at runtime — the grammar's `jointType` rule
is just `ID`, so **no grammar change is needed**. The touch points are:

1. Add a value to the `JointType` enum.
2. Add a record implementing the sealed `Joint` interface, alongside the
   existing six. Implement its geometry inference, validation, and BOM
   contribution — copy the closest existing record (`Dado` if it cuts the
   receiver, `PocketScrew` if it is fastener-based).
3. Map the new name to your record in `CommandVisitor`'s join handling.
4. Cut-list and BOM output follow from the record's own methods.

### Add a template

No Java needed — write a `define … end define` block and drop the `.cds` file
under `src/main/resources/templates/`. See
[CONTRIBUTING.md](../CONTRIBUTING.md#templates--the-easiest-place-to-start) and
the [templates section of SCRIPTING.md](SCRIPTING.md#templates).
